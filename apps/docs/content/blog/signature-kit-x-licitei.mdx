[Licitei](https://licitei.com.br) is one of the largest public-procurement (*licitação*) platforms in Brazil. To bid, a user has to sign: every proposal (*proposta*) and every qualification declaration (*declaração de habilitação*) needs a real digital signature with the bidder's A1 certificate.

That sounds solved — plug in an e-signature API and move on. Except the economics don't work.

## The cost of a signature

Paid e-signature services price **per signature**. For one document that's fine. But a *licitação* is never one document: a bidder signs **dozens of documents at once**, every single time they enter a bid. Multiply per-signature pricing by dozens of documents, by thousands of bidders, by every bid they file, and the line item stops being a feature cost — it becomes the product's cost.

The obvious escape — "let the user sign somewhere else and upload the result" — is worse. It breaks the flow: the bidder leaves Licitei mid-bid, wrestles with another tool, comes back, re-uploads. In a process where bids close at a fixed minute, that friction loses bids.

## Why "A1 signing" still costs so much

Here's the part that's easy to miss. Most services that *do* support A1 certificates still charge heavily for it, and the reason is architectural: they sign **on a server**. The user hands over the `.pfx` (or a managed key vault holds it) along with the document; the service signs server-side and returns the result.

Server-side signing means infrastructure to run, a private key that leaves the user's machine, liability to carry, and a meter to bill against. The per-signature price isn't paying for the cryptography — an RSA signature over a hash is microseconds. It's paying for the *server*.

So Montte and Licitei asked the obvious question: what if the server isn't in the loop at all?

## Sign in the browser

`@signature-kit` runs the **entire signing flow in the browser**. The PDF bytes, the A1 `.p12`/`.pfx` container, and its password stay in the page's local memory and never make a network request. PAdES is computed client-side; the app gets back finished, signed bytes to download or store.

```package-install
@signature-kit/react @signature-kit/pdf @signature-kit/a1
```

<Callout type="error" title="Nothing leaves the device">
  The document, the `.pfx`, and the password live only in the browser tab. The password is wrapped as `Redacted`
  before it reaches `a1SignaturesLayer`, so it never lands in a log or an error message. No request is made — the
  private key never travels.
</Callout>

## The flow

<Steps>
  <Step title="Read each uploaded PDF as bytes">
    `readBrowserFileBytes` turns a `File`/`Blob` into a `Uint8Array`, all in memory. Failures are typed
    (`react.FILE_READ_FAILED`), never a thrown exception.

    ```ts title="read.ts"
    import { readBrowserFileBytes } from "@signature-kit/react/browser-pdf"

    const bytes = await Effect.runPromise(readBrowserFileBytes(file))
    ```
  </Step>

  <Step title="Build the template and place the signature">
    `createBrowserPdfSignatureBuilderState` loads the PDF's pages with pdf-lib and produces a validated template with
    the signature field positioned — explicitly, or auto-placed. Its `template` is what the signer consumes.

    ```ts title="build.ts"
    import { createBrowserPdfSignatureBuilderState } from "@signature-kit/react/browser-pdf"

    const state = await Effect.runPromise(
      createBrowserPdfSignatureBuilderState({
        id: "proposta",
        name: file.name,
        documentId: "proposta",
        documentName: file.name,
        pdf: bytes,
        role: { id: "signer-1", label: "Licitante", required: true },
        draft: { id: "sig", type: "signature", roleId: "signer-1", width: 180, height: 48 },
        placement: { pageIndex: 0, x: 360, y: 96, anchor: "center" },
      }),
    )
    ```
  </Step>

  <Step title="Sign every bid document in one batch">
    `signBrowserPdfBatch` signs the whole set under **one** `a1SignaturesLayer` — the certificate is unlocked once and
    reused for every document. It signs strictly one-by-one, and a document that fails is captured as
    `{ ok: false, error }` instead of aborting the run, so you always get one result per input, in order. `onItemSettled`
    fires as each finishes, so the UI can stream progress and hand each signed file straight back to the bid.

    ```ts title="sign-batch.ts"
    import { signBrowserPdfBatch } from "@signature-kit/react/browser-pdf"
    import { a1SignaturesLayer } from "@signature-kit/a1/signer"
    import { Effect, Redacted } from "effect"

    // `items` = one { id, input } per uploaded document, built with the step above.
    const results = await Effect.runPromise(
      signBrowserPdfBatch(items, {
        onItemSettled: (result, index, total) => {
          if (result.ok) attachToBid(result.id, result.signedPdf)
          else markFailed(result.id, result.error.message)
        },
      }).pipe(
        // The certificate is provided ONCE, for the whole batch, and never leaves the tab.
        Effect.provide(a1SignaturesLayer({ pfx, password: Redacted.make(password) })),
      ),
    )
    ```
  </Step>
</Steps>

That is the whole thing: dozens of documents, one certificate, one pass — and not a single byte sent to a server.

<Callout type="info" title="It still meets the Brazilian requirements">
  The output is PAdES. Need the *rubrica em todas as páginas*? Stamp it before signing with `stampPdfRubric({ pages: "all" })`,
  so a single signature's byte range covers every page. Need the ICP-Brasil policy embedded? Pass `icpBrasil` to the
  signing input. None of it changes where the signing happens — still the browser.
</Callout>

## What this changed for Licitei

- **Zero marginal cost per signature.** The expensive part was the server, and there is no server. Signing dozens of
  documents per bid costs the same as signing one: nothing.
- **The certificate never leaves the device.** The `.pfx` and its password stay in the tab, `Redacted` end to end.
  That is the strongest possible answer to "where did my private key go?" — it didn't go anywhere.
- **It stays in the flow.** The bidder never leaves Licitei. Upload, place, sign the batch, attach — inside the same
  screen, against the same deadline.
- **One bad document doesn't sink the batch.** A failed signature is a typed result, not an exception. The other
  documents still sign; the failure surfaces with a code you can act on.

The cryptography was never the hard part. Moving it off the server — and keeping it typed, batched, and in the browser —
is what made digital-certificate signing something Licitei could simply *include*, rather than meter.
