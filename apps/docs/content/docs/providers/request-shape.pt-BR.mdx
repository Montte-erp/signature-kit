APIs de provedores criam cerimônias de assinatura no upstream. Elas não são backends locais de `Signatures`: o provedor possui links, notificações, destinatários, envelopes e UX de auditoria. O SignatureKit mantém as chamadas boring: input tipado, credenciais Redacted, `SignatureHttpClient` e `SignatureKitError` em falhas recuperáveis.

```package-install
@signature-kit/core @signature-kit/clicksign @signature-kit/assinafy @signature-kit/zapsign @signature-kit/docuseal @signature-kit/documenso
```

## Shape compartilhado

Todo provider começa dos mesmos fatos de request do SignatureKit: `title`, `documents`, `recipients`, `subject`, `message`, `send`, `expiresAt` e `redirectUrl` opcionais. Cada página de provider explica como esses fatos mapeiam para seu protocolo upstream.

<Callout type="info" title="Local vs remoto">
  Use `@signature-kit/pdf` + `Signatures` quando o SignatureKit deve assinar bytes. Use funções de request de provider quando uma plataforma upstream possui a cerimônia.
</Callout>
