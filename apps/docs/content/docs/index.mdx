SignatureKit is an Effect-native signing library for TypeScript. You define one `SignerAdapter` — where the power to sign comes in — and the rest plugs into it: XML-DSig, PDF/PAdES ICP-Brasil, an in-browser A1 demo, and remote providers like DocuSign. Swapping the backend behind the boundary never touches your signing code, and every failure is a typed `SignatureKitError`, not a thrown exception.

```text
                        Application
                            |
                            v
              +---------------------------+
              |    Signatures service     |
              |  inspect / sign / verify  |
              +---------------------------+
                            |
              ---- one boundary: SignerAdapter ----
              |             |             |
              v             v             v
         A1 / PKCS#12    Formats       Remote
         (browser /      XML-DSig      providers
          server)        PDF/PAdES     (DocuSign, ...)
```

You provide the signer as a `Layer`. The `Signatures` service exposes `inspect`, `sign`, and `verify` — the program below never knows whether the key came from the browser, the server, or anywhere else.

```ts title="signature-kit.ts"
import { a1SignaturesLayer } from "@signature-kit/a1/signer"
import { signatures } from "@signature-kit/core/signatures"
import { Effect, Redacted } from "effect"

export const program = Effect.gen(function* () {
  const identity = yield* signatures.inspect()
  const artifact = yield* signatures.sign({ content, algorithm: "rsa-sha256" })
  return { identity, artifact }
}).pipe(
  Effect.provide(
    a1SignaturesLayer({
      pfx,
      password: Redacted.make("certificate-password"),
    }),
  ),
)
```

## Why SignatureKit

- **One boundary, many formats.** `signatures.sign`, `signXml`, and `signPdf` all run over the same `SignerAdapter`. Swap the argument of `signaturesLayer` and the signing job stays identical.
- **A1 in the browser or on the server.** The same e-CPF / e-CNPJ PKCS#12 signer runs in a Node service or directly in the browser — try the live in-browser signer on the [browser signing](/docs/components/browser-signing) page.
- **No provider gateway.** SignatureKit does not normalize remote signers into one interface. DocuSign, Clicksign, Assinafy, ZapSign, DocuSeal, Adobe Acrobat Sign, Dropbox Sign, and Documenso are each their own package and function, sharing only the Effect shape.
- **A typed error channel.** One `SignatureKitError` with eighteen literal codes flows through every call, so failures are values you match on, never exceptions you catch by accident.
- **Effect-native, Redacted secrets.** Everything is an `Effect`; certificate passwords and provider keys are `Redacted.make(...)`, so secrets never land in logs or stack traces.

## What it is not

SignatureKit is not a hosted e-signature SaaS, not a provider-normalizing gateway that pretends every remote signer has the same API, and not a PKI or certificate authority. It is the signing boundary apps keep rebuilding — adapter setup, one consistent `inspect` / `sign` / `verify` call, formats that reuse it, typed errors, and a clean seam where you swap the backend without rewriting the document logic.

```package-install
@signature-kit/core @signature-kit/a1
```

<Callout type="warn" title="Local checks are not legal validity">
  `verifyXml`, `verifyPdf`, and `signatures.verify` confirm the cryptography against the key in the
  document — not legal validity. Verify with your own ICP-Brasil trust chain and policy before
  treating a signed document as valid.
</Callout>
