`@signature-kit/react` é a camada React do SignatureKit. Ela desenha a _intenção_ de assinatura — quem assina, o quê e onde — e prepara o documento para a camada de assinatura. A chave nunca passa por aqui: a assinatura criptográfica acontece em `@signature-kit/pdf` / `@signature-kit/a1`, atrás de `Signatures`. Esta página é o mapa do pacote; cada componente, hook e função tem sua própria página de referência.

```package-install
@signature-kit/react @signature-kit/pdf @signature-kit/a1
```

## Os cinco pontos de entrada

O pacote não tem um barrel único: cada arquivo é seu próprio caminho de import, então você carrega apenas o que usa (e o renderer `@react-pdf/renderer`, por exemplo, fica isolado em `react-pdf`).

| Ponto de entrada | Import | O que entrega |
| ----------- | ------ | ---------------- |
| `config` | `@signature-kit/react/config` | Schemas e tipos do Effect (`ReactSignatureTemplate`, `ReactSignatureField`, `ReactSignatureFieldDraft`, `ReactSignatureFieldPlacement`…), os helpers literais `*Value` e a classe `ReactIntegrationError`. |
| `builder` | `@signature-kit/react/builder` | Funções puras sobre um template: `createReactSignatureTemplate`, `placeReactSignatureField`, `autoPlaceReactSignatureField`, `moveReactSignatureField`, `removeReactSignatureField`, `assignReactSignatureFieldValue` e similares — todas retornam `Effect`. |
| `components` | `@signature-kit/react/components` | UI: `SignatureBuilderSurface` (+ subpartes), `SignatureInkPad`, a store (`createSignatureBuilderStore` / `useSignatureBuilderStore`) e os hooks de seleção. |
| `browser-pdf` | `@signature-kit/react/browser-pdf` | Fluxo de assinatura no navegador: `loadBrowserPdfDocument`, `createBrowserPdfSignatureBuilderState`, `signBrowserPdf` e a fila `useBrowserPdfSigningQueue`. |
| `react-pdf` | `@signature-kit/react/react-pdf` | `ReactPdfSignatureTemplateDocument` e `renderReactSignatureTemplatePdf` para rasterizar um template em bytes de PDF via `@react-pdf/renderer`. |

## O modelo: template → roles → fields → store

Tudo no pacote gira em torno de um `ReactSignatureTemplate`. Ele é a fonte da verdade imutável, e os quatro níveis se encaixam assim:

- **template** — `id`, `name`, mais `documents`, `roles` e `fields`. Cada documento tem `pages` com `width`/`height` em pontos de PDF.
- **roles** — os `ReactSignerRole` (signatários): `id`, `label`, `email?`, `required?`. Todo field aponta para um `roleId`.
- **fields** — o `ReactSignatureField` posicionado: `type` (`signature` | `initials` | `text` | `date` | `checkbox`), `documentId`, `roleId`, um `rect` (`pageIndex`, `x`, `y`, `width`, `height`) e um `value?` opcional. Para _criar_ um field você passa um `ReactSignatureFieldDraft` (sem `rect`, com `width`/`height`) dentro de um `ReactSignatureFieldPlacement`.
- **store** — a `SignatureBuilderStore` (TanStack Store) envolve o template com estado de UI (`selectedFieldId`, `draft`) e expõe operações que retornam `Effect`: `placeField`, `moveField`, `removeField`, `assignValue`, `selectField`, `setDraft`.

```tsx title="ContractBuilder.tsx"
import { Effect } from "effect"
import {
  SignatureBuilderSurface,
  useSignatureBuilderStore,
} from "@signature-kit/react/components"
import type { ReactSignatureBuilderState } from "@signature-kit/react/config"

function ContractBuilder({ initialState }: { initialState: ReactSignatureBuilderState }) {
  const store = useSignatureBuilderStore(initialState)

  return (
    <SignatureBuilderSurface
      store={store}
      onFieldPlacement={(placement) => void Effect.runPromise(store.placeField(placement))}
    />
  )
}
```

<Callout type="info" title="Por que um draft, e não um field">
  A surface entrega um `ReactSignatureFieldPlacement` (documento, página, ponto `x`/`y`, `draft` e `anchor`). O builder
  resolve o `rect` final a partir do `anchor` e _faz o clamp_ do field aos limites da página antes de validar. Você descreve a
  intenção; o builder produz o `ReactSignatureField`.
</Callout>

## A camada Effect

Nenhuma função do pacote executa `Effect` por baixo dos panos. Toda operação — validar um template, posicionar um field, carregar um PDF, assinar — retorna um `Effect.Effect<…, ReactIntegrationError, …>`, e **é o seu app que decide quando executá-la**, tipicamente com `Effect.runPromise` no event handler do React. É nesse ponto que um erro vira UX.

```tsx title="handler.tsx"
import { Effect } from "effect"

const onPlace = async (placement: ReactSignatureFieldPlacement) => {
  const result = await Effect.runPromiseExit(store.placeField(placement))
  if (result._tag === "Failure") {
    // ReactIntegrationError: code (e.g. "react.FIELD_OUT_OF_BOUNDS"), reason, retryable
    return showPlacementError(result.cause)
  }
}
```

Todo erro do pacote é um único `ReactIntegrationError` (erro tagged) com `code` (`react.INVALID_BUILDER_INPUT`, `react.FIELD_OUT_OF_BOUNDS`, `react.NO_AVAILABLE_PLACEMENT`, `react.PDF_LOAD_FAILED`, `react.QUEUE_REJECTED`…), `retryable: boolean` e um `reason` legível. `signBrowserPdf` é o único cujo `Effect` exige o serviço `Signatures` no contexto — todo o resto roda sem dependências.

<Callout type="info" title="runPromise vs runPromiseExit">
  Use `Effect.runPromiseExit` quando quiser ler `code` e `retryable` para montar a mensagem certa; `Effect.runPromise`
  rejeita a Promise com o erro e basta quando um `try/catch` já dá conta.
</Callout>

## O estilo data-slot

Os componentes visuais (`SignatureBuilderSurface` e subpartes, `SignatureInkPad`) seguem o padrão data-slot: cada elemento expõe um `data-slot="…"` estável e estilos inline mínimos. Você estiliza por atributo, sem alterar a lógica nem fazer fork do componente:

```css title="builder.css"
[data-slot="signature-builder-page"] {
  box-shadow: var(--shadow-md);
}
[data-slot="signature-builder-field"][aria-pressed="true"] {
  outline: 2px solid var(--ring);
}
```

Cada componente também aceita `className` / `style` na raiz e nas partes relevantes, de modo que classes utilitárias convivem com a estilização baseada em slots. A anatomia completa de slots de cada componente fica na sua própria página.
