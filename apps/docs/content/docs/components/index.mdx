`@signature-kit/react` is now a thin browser/React layer over SignatureKit's Effect runtime. The launch scope is intentionally narrow: **PDF signing with an A1 / PKCS#12 certificate in the browser**. Remote providers, DocuSeal builder bridges, generic field types, queue hooks, and `react-pdf` rendering are outside this React package surface.

```package-install
@signature-kit/react @signature-kit/a1 @signature-kit/pdf
```

## Public shape

| Import | Use |
| --- | --- |
| `@signature-kit/react/browser-pdf` | Read a browser `File`, derive PDF page geometry, build an A1 signature placement, and sign one or more PDFs with PAdES. |
| `@signature-kit/react/components` | Headless placement UI (`SignatureBuilderSurface`) plus store/selectors for the A1 demo. Store writes that change templates return `Effect`. |
| `@signature-kit/react/builder` | Pure Effect functions for the A1 signature template: validate, place/move/remove one `signature` field, and derive the PDF widget appearance. |
| `@signature-kit/react/config` | Schemas, derived types, and `ReactIntegrationError`. |

## Runtime rule

React does not own signing. Browser code reads files and handles events; Effect owns the workflow; the A1 signer is provided at the app boundary:

```ts title="sign.ts"
import { a1SignaturesLayer } from "@signature-kit/a1/signer"
import { signBrowserPdf } from "@signature-kit/react/browser-pdf"
import { Effect, Redacted } from "effect"

const signed = await Effect.runPromise(
  signBrowserPdf({ pdf, template, fieldId: "sig" }).pipe(
    Effect.provide(a1SignaturesLayer({ pfx, password: Redacted.make(password) })),
  ),
)
```

No hidden `runPromise`, no hidden `Effect.provide`, no generic signer UX in the React package.
