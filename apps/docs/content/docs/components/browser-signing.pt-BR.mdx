`@signature-kit/react/browser-pdf` é a costura A1 no navegador: lê um `File`/`Blob`, deriva as dimensões das páginas do PDF, posiciona um campo `signature` e assina com PAdES sob `a1SignaturesLayer`. Os bytes do PDF, o `.p12`/`.pfx` e a senha ficam na memória do navegador.

```package-install
@signature-kit/react @signature-kit/a1 @signature-kit/pdf
```
## Instalação via registry

```bash
npx shadcn@latest add a1-browser-signing --registry {URL_DA_DOCS}/registry
```

Instale o bloco da UI sem depender do app de docs: ele é copiado para `components/a1-browser-signing.tsx`. O item da registry puxa os primitivos shadcn `button`, `input` e `label` mais as dependências locais de formulário/store do TanStack; SignatureKit continua limitado a `@signature-kit/react`, `@signature-kit/a1` e `@signature-kit/pdf`.

O bloco instalável segue a ideia que gostamos no PDFX: manter a UI de PDF como um componente copiável no estilo shadcn, enquanto a primitiva de assinatura continua sendo a API Effect do SignatureKit.


<Callout type="error" title="Nada sai do dispositivo">
  `readBrowserFileBytes`, `createBrowserPdfSignatureBuilderState`, `signBrowserPdf` e `signBrowserPdfBatch` não enviam documento nem certificado. Envolva a senha com `Redacted` antes de fornecer a camada A1.
</Callout>

## Fluxo mínimo

```ts title="a1-browser-signing.ts"
import { a1SignaturesLayer } from "@signature-kit/a1/signer"
import {
  createBrowserPdfSignatureBuilderState,
  readBrowserFileBytes,
  signBrowserPdf,
} from "@signature-kit/react/browser-pdf"
import { Effect, Redacted } from "effect"

const bytes = await Effect.runPromise(readBrowserFileBytes(file))

const state = await Effect.runPromise(
  createBrowserPdfSignatureBuilderState({
    id: "contract-a1",
    name: file.name,
    documentId: "contract",
    documentName: file.name,
    pdf: bytes,
    role: { id: "signer", label: "Signer A1", required: true },
    draft: { id: "sig", type: "signature", roleId: "signer", width: 156, height: 42 },
    autoPlacement: { page: "last", slot: "bottom-right", margin: 24 },
  }),
)

const signed = await Effect.runPromise(
  signBrowserPdf({
    pdf: bytes,
    template: state.template,
    fieldId: "sig",
    reason: "Signed locally with A1",
    signatureLength: 16384,
    policy: "pades-ades",
  }).pipe(
    Effect.provide(a1SignaturesLayer({ pfx, password: Redacted.make(password) })),
  ),
)
```

## Duas pessoas, dois certificados A1

`signBrowserPdfBatch` serve para muitos PDFs sob uma única camada A1 fornecida. Quando cada pessoa tem um certificado diferente, mantenha o mesmo fluxo de PDF/template e forneça cada signatário separadamente na borda da ação do navegador.

```ts title="two-browser-a1-signers.ts"
const mariaSignedPdf = await Effect.runPromise(
  signBrowserPdf({
    pdf: bytes,
    template: state.template,
    fieldId: "sig",
    reason: "Maria aprovou no navegador",
    name: "Maria Souza",
  }).pipe(
    Effect.provide(
      a1SignaturesLayer({
        pfx: mariaPfx,
        password: Redacted.make(mariaPassword),
      }),
    ),
  ),
)

const joaoSignedPdf = await Effect.runPromise(
  signBrowserPdf({
    pdf: bytes,
    template: state.template,
    fieldId: "sig",
    reason: "João aprovou no navegador",
    name: "João Silva",
  }).pipe(
    Effect.provide(
      a1SignaturesLayer({
        pfx: joaoPfx,
        password: Redacted.make(joaoPassword),
      }),
    ),
  ),
)
```

Cada programa produz um PDF assinado separado. A superfície React continua responsável apenas por leitura de arquivos no navegador, posicionamento e ligação da ação; a assinatura A1 fica na camada Effect que você fornece.


## API

### `readBrowserFileBytes`

```ts
readBrowserFileBytes(file: Blob): Effect.Effect<Uint8Array, ReactIntegrationError>
```

Lê um arquivo/blob do navegador como bytes. Falha com `react.FILE_READ_FAILED`.

### `createBrowserPdfSignatureBuilderState`

```ts
createBrowserPdfSignatureBuilderState(input): Effect.Effect<ReactSignatureBuilderState, ReactIntegrationError>
```

Carrega o PDF com `pdf-lib`, cria um template com um signer role e opcionalmente posiciona a assinatura. Passe `placement` ou `autoPlacement`, nunca os dois.

### `signBrowserPdf`

```ts
signBrowserPdf(input): Effect.Effect<Uint8Array, ReactIntegrationError | PdfError | CmsError | SignatureKitError, Signatures>
```

Valida `BrowserPdfSigningInput`, deriva o retângulo do widget PDF a partir de `template + fieldId` e chama `signPdf`. Exige `Signatures`; no pacote React, forneça `a1SignaturesLayer` na borda da aplicação.

### `signBrowserPdfBatch`

```ts
signBrowserPdfBatch(
  items: ReadonlyArray<BrowserPdfSigningBatchItem>,
  callbacks?: { onItemSettled?: (result: BrowserPdfBatchResult, index: number, total: number) => void },
): Effect.Effect<ReadonlyArray<BrowserPdfBatchResult>, never, Signatures>
```

Assina em sequência sob uma única camada A1 fornecida. Cada item retorna `{ ok: true, signedPdf }` ou `{ ok: false, error }`; um documento com falha não aborta os demais.

## Erros

`ReactIntegrationError` é um erro tipado no canal do Effect. Códigos comuns aqui: `react.INVALID_BUILDER_INPUT`, `react.EMPTY_TEMPLATE`, `react.UNKNOWN_FIELD`, `react.FIELD_OUT_OF_BOUNDS`, `react.NO_AVAILABLE_PLACEMENT`, `react.FILE_READ_FAILED` e `react.PDF_LOAD_FAILED`.
