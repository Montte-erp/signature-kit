title: Browser signing
description: "Load, place, queue and sign PDFs with A1 entirely in the browser, via @signature-kit/react/browser-pdf — bytes and password never leave the device."
---

The `@signature-kit/react/browser-pdf` subpath gathers the functions that run the **entire signing flow in the browser**: reading a `File`/`Blob`, assembling the builder from a loaded PDF, signing with PAdES, and queuing signatures with TanStack Pacer. Each function returns a typed `Effect` — the app decides when to run it and where to provide the `Signatures` boundary (e.g. `a1SignaturesLayer`).

```package-install
@signature-kit/react @signature-kit/pdf @signature-kit/a1
```

<Callout type="error" title="Everything happens in the browser — nothing goes to the server">
  The PDF bytes, the A1 `.p12`/`.pfx` container and the password stay in the browser's local memory. The password is wrapped as
  `Redacted` before reaching `a1SignaturesLayer`, so it never leaks into logs or error messages. None of these functions
  make a network request: the private key and the document never leave the signer's device.
</Callout>

## Live demo

The React island below runs the full flow in your browser: upload one or more PDFs, place each signature, add your A1 certificate with its password, and sign the whole batch with PAdES — optionally repeating the rubric on every page. It signs sequentially with [`signBrowserPdfBatch`](#signbrowserpdfbatch) under a single `Signatures` layer; the bytes and password never leave the page.

## Flow

<Steps>
  <Step title="Read the file as bytes">
    `readBrowserFileBytes` takes a `Blob` (or `File`) and returns a `Uint8Array`. On failure, the error is
    `ReactIntegrationError` with code `react.FILE_READ_FAILED` (retryable).

    ```ts title="read.ts"
    import { readBrowserFileBytes } from "@signature-kit/react/browser-pdf"
    import { Effect } from "effect"

    const onFile = (file: File) =>
      Effect.runPromise(readBrowserFileBytes(file))
    ```
  </Step>

  <Step title="Assemble the builder from the loaded PDF">
    `createBrowserPdfSignatureBuilderState` validates the input, loads the PDF pages (`loadBrowserPdfDocument`), creates the
    template and — optionally — places the field (`placement`) or auto-places it (`autoPlacement`). It accepts **one or the other**,
    never both (otherwise it fails with `react.INVALID_BUILDER_INPUT`). The result is a `ReactSignatureBuilderState` ready for
    `builderStore.setState(...)`.

    ```ts title="builder.ts"
    import { createBrowserPdfSignatureBuilderState } from "@signature-kit/react/browser-pdf"

    const state = createBrowserPdfSignatureBuilderState({
      id: "uploaded-template",
      name: file.name,
      documentId: "uploaded-pdf",
      documentName: file.name,
      pdf: bytes,
      role: { id: "signer-1", label: "A1 signer", required: true },
      draft: { id: "sig", type: "signature", roleId: "signer-1", width: 156, height: 42 },
    })
    ```
  </Step>

  <Step title="Sign with PAdES">
    `signBrowserPdf` validates the `BrowserPdfSigningInput`, derives the appearance from the `fieldId` in the `template` and calls
    `signPdf`. It requires the `Signatures` boundary in the `Effect` context — provide it with `a1SignaturesLayer({ pfx, password })`.

    ```ts title="sign.ts"
    import { signBrowserPdf } from "@signature-kit/react/browser-pdf"
    import { a1SignaturesLayer } from "@signature-kit/a1/signer"
    import { Effect, Redacted } from "effect"

    const signed = signBrowserPdf({
      pdf: bytes,
      template,
      fieldId: "sig",
      name: "Maria de Souza",
      location: "Browser",
      signatureLength: 16384,
    }).pipe(
      Effect.provide(a1SignaturesLayer({ pfx, password: Redacted.make(password) })),
    )
    ```
  </Step>

  <Step title="Sign a batch sequentially">
    `signBrowserPdfBatch` is the batch primitive the demo above uses. Pass an array of
    `{ id, input }` items and provide **one** `Signatures` layer for the whole run. It signs strictly
    one-by-one (`concurrency: 1`), and a document that fails is captured as `{ ok: false, error }`
    — it never aborts the rest, so you always get one result per input, in order. No `start()`/`flush()`.

    ```ts title="batch.ts"
    import { signBrowserPdfBatch } from "@signature-kit/react/browser-pdf"
    import { a1SignaturesLayer } from "@signature-kit/a1/signer"
    import { Effect, Redacted } from "effect"

    const results = await Effect.runPromise(
      signBrowserPdfBatch(
        [
          { id: "contract.pdf", input: contractInput },
          { id: "annex.pdf", input: annexInput },
        ],
        {
          onItemSettled: (result, index, total) => {
            if (result.ok) download(result.id, result.signedPdf)
            else console.error(result.id, result.error.message)
          },
        },
      ).pipe(
        Effect.provide(a1SignaturesLayer({ pfx, password: Redacted.make(password) })),
      ),
    )
    ```
  </Step>
</Steps>

<Callout type="info" title="Alternative: a streaming queue">
  Prefer a long-lived, push-as-you-go queue over a one-shot array? `createBrowserPdfSigningQueue`
  (and the `useBrowserPdfSigningQueue` hook) wrap `AsyncQueuer` (TanStack Pacer) with a reactive
  snapshot — see [createBrowserPdfSigningQueue](#createbrowserpdfsigningqueue) below. For "load several
  documents and sign them all", `signBrowserPdfBatch` is simpler and needs no `start()`/`flush()`.
</Callout>

## API

### readBrowserFileBytes

Reads a `Blob`/`File` as a `Uint8Array`.

```ts
readBrowserFileBytes(
  file: Blob,
): Effect.Effect<Uint8Array, ReactIntegrationError>
```

### loadBrowserPdfDocument

Decodes a `BrowserPdfDocumentInput`, opens the PDF with `pdf-lib` and returns a `ReactSignatureDocument` with its pages (dimensions and label). Fails with `react.EMPTY_TEMPLATE` if the PDF has no pages, or `react.PDF_LOAD_FAILED` if it cannot be parsed.

```ts
loadBrowserPdfDocument(
  input: BrowserPdfDocumentInput,
): Effect.Effect<ReactSignatureDocument, ReactIntegrationError>
```

### signBrowserPdfBatch

Signs many PDFs one-by-one under a single provided `Signatures` layer — the demo's batch path. Each item runs through `signBrowserPdf`; a failure is captured as `{ ok: false, error }` and never aborts the run, so the result array holds exactly one entry per input, in order.

```ts
signBrowserPdfBatch(
  items: ReadonlyArray<{ id: string; input: BrowserPdfSigningInput }>,
  callbacks?: { onItemSettled?: (result: BrowserPdfBatchResult, index: number, total: number) => void },
): Effect.Effect<ReadonlyArray<BrowserPdfBatchResult>, never, Signatures>
```

`BrowserPdfBatchResult` is `{ id; ok: true; signedPdf: Uint8Array }` or `{ id; ok: false; error }`. Provide the layer once for the whole batch: `.pipe(Effect.provide(a1SignaturesLayer({ pfx, password })))`.

### createBrowserPdfSigningQueue

Creates a signing queue backed by `AsyncQueuer` (TanStack Pacer). An alternative to `signBrowserPdfBatch` for streaming, push-as-you-go signing.

```ts
createBrowserPdfSigningQueue(
  executor: BrowserPdfSigningQueueExecutor,
  options?: BrowserPdfSigningQueueOptions,
  callbacks?: BrowserPdfSigningQueueCallbacks,
): Effect.Effect<BrowserPdfSigningQueue, ReactIntegrationError>
```

`executor` is `(input: BrowserPdfSigningInput) => Promise<Uint8Array>`.

`options` — `BrowserPdfSigningQueueOptions`:

<TypeTable
  type={{
    concurrency: { type: "number", description: "Concurrent signatures." },
    waitMillis: { type: "number", description: "Wait between processings (maps to Pacer's wait)." },
    maxSize: { type: "number", description: "Maximum queue capacity; items beyond it are rejected." },
    started: { type: "boolean", description: "Starts the queue already processing." },
  }}
/>

`callbacks` — `BrowserPdfSigningQueueCallbacks`:

<TypeTable
  type={{
    onSuccess: { type: "(success: BrowserPdfSigningQueueSuccess, item: BrowserPdfSigningQueueItem) => void", description: "Item signed successfully. success = { id, signedPdf }." },
    onError: { type: "(error: Error, item: BrowserPdfSigningQueueItem) => void", description: "Failed to sign the item." },
  }}
/>

The returned `BrowserPdfSigningQueue` exposes `getSnapshot`, `subscribe`, `add`, `start`, `stop`, `clear`, `flush` and `abort`. `add(item: BrowserPdfSigningQueueItem)` validates the item and returns an `Effect` that fails with `react.QUEUE_REJECTED` when the queue does not accept the item.

<Callout type="info" title="In React, use the hook">
  `useBrowserPdfSigningQueue(executor, options?, callbacks?)` returns a `BrowserPdfSigningQueueController` — the same API
  plus `snapshot`, a reactive `BrowserPdfSigningQueueSnapshot` (`status`, `pendingCount`, `activeCount`,
  `successCount`, `errorCount`, `settledCount`, `rejectionCount`). Reach for the queue when you need a long-lived,
  streaming pipeline; the demo above instead uses [`signBrowserPdfBatch`](#signbrowserpdfbatch) to sign a fixed set of documents in one pass.
</Callout>

## Errors

All functions fail with `ReactIntegrationError` in the `Effect` error channel — never a loose exception. Relevant codes for this subpath:

- [`react.FILE_READ_FAILED`](/docs/errors) — the `Blob`/`File` could not be read as an `ArrayBuffer` (retryable).
- [`react.PDF_LOAD_FAILED`](/docs/errors) — the loaded PDF could not be parsed.
- [`react.EMPTY_TEMPLATE`](/docs/errors) — the PDF has no pages to place the signature on.
- [`react.INVALID_BUILDER_INPUT`](/docs/errors) — input failed schema validation (e.g. `placement` and `autoPlacement` together).
- [`react.QUEUE_REJECTED`](/docs/errors) — the queue refused the item (e.g. `maxSize` reached; retryable).
