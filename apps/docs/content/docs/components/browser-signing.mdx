`@signature-kit/react/browser-pdf` is the A1 browser seam: read a `File`/`Blob`, derive PDF page dimensions, place one `signature` field, and sign with PAdES under `a1SignaturesLayer`. The PDF bytes, `.p12`/`.pfx`, and password stay in browser memory.

```package-install
@signature-kit/react @signature-kit/a1 @signature-kit/pdf
```
## Registry install

```bash
npx shadcn@latest add a1-browser-signing --registry {DOCS_ORIGIN}/registry
```

This installs the UI block (without docs-app extras) to `components/a1-browser-signing.tsx`. The registry item pulls `button`, `input`, and `label` shadcn primitives plus its local TanStack form/store dependencies; SignatureKit stays limited to `@signature-kit/react`, `@signature-kit/a1`, and `@signature-kit/pdf`.

The installable block follows the same idea we liked in PDFX: keep the PDF-facing UI as a copied shadcn-style component, while the signing primitive remains SignatureKit's Effect API.


<Callout type="error" title="Nothing leaves the device">
  `readBrowserFileBytes`, `createBrowserPdfSignatureBuilderState`, `signBrowserPdf`, and `signBrowserPdfBatch` do not upload the document or the certificate. Wrap the password with `Redacted` before providing the A1 layer.
</Callout>

## Minimal flow

```ts title="a1-browser-signing.ts"
import { a1SignaturesLayer } from "@signature-kit/a1/signer"
import {
  createBrowserPdfSignatureBuilderState,
  readBrowserFileBytes,
  signBrowserPdf,
} from "@signature-kit/react/browser-pdf"
import { Effect, Redacted } from "effect"

const bytes = await Effect.runPromise(readBrowserFileBytes(file))

const state = await Effect.runPromise(
  createBrowserPdfSignatureBuilderState({
    id: "contract-a1",
    name: file.name,
    documentId: "contract",
    documentName: file.name,
    pdf: bytes,
    role: { id: "signer", label: "A1 signer", required: true },
    draft: { id: "sig", type: "signature", roleId: "signer", width: 156, height: 42 },
    autoPlacement: { page: "last", slot: "bottom-right", margin: 24 },
  }),
)

const signed = await Effect.runPromise(
  signBrowserPdf({
    pdf: bytes,
    template: state.template,
    fieldId: "sig",
    reason: "Signed locally with A1",
    signatureLength: 16384,
    policy: "pades-ades",
  }).pipe(
    Effect.provide(a1SignaturesLayer({ pfx, password: Redacted.make(password) })),
  ),
)
```

## Two people, two A1 certificates

`signBrowserPdfBatch` is for many PDFs under one provided A1 layer. When each person has a different certificate, keep the PDF/template flow the same and provide each signer separately at the browser action boundary.

```ts title="two-browser-a1-signers.ts"
const mariaSignedPdf = await Effect.runPromise(
  signBrowserPdf({
    pdf: bytes,
    template: state.template,
    fieldId: "sig",
    reason: "Maria approved in-browser",
    name: "Maria Souza",
  }).pipe(
    Effect.provide(
      a1SignaturesLayer({
        pfx: mariaPfx,
        password: Redacted.make(mariaPassword),
      }),
    ),
  ),
)

const joaoSignedPdf = await Effect.runPromise(
  signBrowserPdf({
    pdf: bytes,
    template: state.template,
    fieldId: "sig",
    reason: "João approved in-browser",
    name: "João Silva",
  }).pipe(
    Effect.provide(
      a1SignaturesLayer({
        pfx: joaoPfx,
        password: Redacted.make(joaoPassword),
      }),
    ),
  ),
)
```

Each program produces a separate signed PDF. The React surface still owns only browser file reads, placement, and action wiring; A1 signing stays in the Effect layer you provide.


## API

### `readBrowserFileBytes`

```ts
readBrowserFileBytes(file: Blob): Effect.Effect<Uint8Array, ReactIntegrationError>
```

Reads a browser file/blob into bytes. Fails with `react.FILE_READ_FAILED`.

### `createBrowserPdfSignatureBuilderState`

```ts
createBrowserPdfSignatureBuilderState(input): Effect.Effect<ReactSignatureBuilderState, ReactIntegrationError>
```

Loads the PDF with `pdf-lib`, creates a template with one signer role, and optionally places the signature. Pass either `placement` or `autoPlacement`, never both.

### `signBrowserPdf`

```ts
signBrowserPdf(input): Effect.Effect<Uint8Array, ReactIntegrationError | PdfError | CmsError | SignatureKitError, Signatures>
```

Validates `BrowserPdfSigningInput`, derives the PDF widget rectangle from `template + fieldId`, and calls `signPdf`. It requires `Signatures`; for the React package, provide `a1SignaturesLayer` at the app boundary.

### `signBrowserPdfBatch`

```ts
signBrowserPdfBatch(
  items: ReadonlyArray<BrowserPdfSigningBatchItem>,
  callbacks?: { onItemSettled?: (result: BrowserPdfBatchResult, index: number, total: number) => void },
): Effect.Effect<ReadonlyArray<BrowserPdfBatchResult>, never, Signatures>
```

Signs sequentially under one provided A1 layer. Each item returns `{ ok: true, signedPdf }` or `{ ok: false, error }`; one failed document does not abort the rest.

## Errors

`ReactIntegrationError` is a typed Effect error. Common codes here: `react.INVALID_BUILDER_INPUT`, `react.EMPTY_TEMPLATE`, `react.UNKNOWN_FIELD`, `react.FIELD_OUT_OF_BOUNDS`, `react.NO_AVAILABLE_PLACEMENT`, `react.FILE_READ_FAILED`, and `react.PDF_LOAD_FAILED`.
